🔄 Recommended Schema Refactor: settings Table

Let’s shift to a key-value model, where each setting is stored as a separate row. This gives you:

  • Flexibility to add/edit settings without altering table structure

  • Easier grouping and retrieval by module or context

  • Cleaner UI rendering and validation

🧱 New Table Structure

sql

Copy

CREATE TABLE settings (
  key_settings INT(10) UNSIGNED AUTO_INCREMENT PRIMARY KEY,
  setting_key VARCHAR(100) NOT NULL,
  setting_value TEXT NOT NULL,
  setting_group VARCHAR(50) DEFAULT 'general',
  setting_type ENUM('text','number','boolean','url','color','json') DEFAULT 'text',
  is_active TINYINT(1) DEFAULT 1,
  entry_date_time TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

setting_value

setting_type

is_active

Toggle for deprecated or draft settings
